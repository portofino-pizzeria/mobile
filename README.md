# Portofino Pizzeria — Mobile App

The customer-facing mobile app for **Portofino Pizzeria**, built with
[Expo](https://expo.dev) (React Native) + TypeScript + Expo Router — iOS,
Android, and web from a single codebase.

## Features

- Browse the menu (loaded from the backend)
- Cart with quantity steppers
- Checkout with **Stripe** or **PayPal** (hosted checkout)
- Order confirmation & live status polling

## Stack

- Expo SDK 56 · React Native · TypeScript (strict)
- Expo Router (file-based routing)
- `react-native-web` (shared web target)
- **UI Bridge** (`@qontinui/ui-bridge-native`) — integrated from the start

## UI Bridge

The app is a UI Bridge project. `src/app/_layout.tsx` wraps everything in
`UIBridgeNativeProvider`, which — in dev builds — runs an embedded control
server on **port 8087** (matching the Qontinui runner's `ui_bridge_port`). This
lets the runner discover and drive the UI **semantically** (by id/label) rather
than by screen coordinates.

- Interactive controls use the `BridgeButton` / `BridgeInput` wrappers in
  `src/components/bridge.tsx` (they call `useUIElement`). Each has a stable
  `uiId` (e.g. `menu-add-margherita`, `pay-stripe`). Buttons pass their press
  into the bridge `handlers.onPress`, so a runner-dispatched press runs the same
  code as a real tap.
- Screens register higher-level semantic actions with `useUIComponent`
  (e.g. `menu.addToCart`, `cart.checkout`, `checkout.payWithStripe`).

In production builds the bridge server/debug features are disabled (`__DEV__`).

### The server transport (important)

`UIBridgeNativeProvider` needs a **`serverAdapter`** to actually open the HTTP
control server — without one it logs *"HTTP server not available"* and 8087
stays closed. We supply one in `src/lib/ui-bridge-server-adapter.ts`, a minimal
HTTP/1.1 server backed by **`react-native-tcp-socket`** (a native module).

Consequences:

- **8087 only runs in a native dev build** (`npx expo run:android` / `run:ios`),
  **not** in Expo Go or on web — Expo Go can't load the tcp-socket native module,
  and web has no TCP server.
- To reach the server from your host against an emulator:
  ```bash
  adb forward tcp:8087 tcp:8087
  curl http://localhost:8087/ui-bridge/health
  curl http://localhost:8087/ui-bridge/control/elements     # registered controls
  curl http://localhost:8087/ui-bridge/control/components   # semantic actions
  ```
- Driving actions over HTTP:
  - Press an element:
    `POST /ui-bridge/control/element/<uiId>/action  {"action":"press"}`
  - Invoke a component action (note the `params` wrapper):
    `POST /ui-bridge/control/component/menu/action/addToCart  {"params":{"itemId":"diavola"}}`

### Running on the Android emulator (Windows)

Prereqs (installed once, env vars persisted): **JDK 17**, the **Android SDK**
(`platform-tools`, `emulator`, `platforms;android-36`, `build-tools;36.0.0`,
`system-images;android-36;google_apis;x86_64`), and an AVD.

```bash
# start the emulator, then:
npx expo run:android          # first build ~10 min; rebuilds are cached
```

`android/` is generated by `expo prebuild` (Continuous Native Generation) — treat
it as build output; re-run prebuild after changing native config/plugins.

## Backend

This app talks to the API in [`../backend`](../backend). Start it first
(`npm run dev` there). The API base URL is derived automatically from Expo's dev
host so a physical phone reaches your machine over LAN; override with
`EXPO_PUBLIC_API_URL` if needed (see `src/lib/config.ts`).

## Get started

1. Install dependencies:

   ```bash
   npm install
   ```

2. Run the app:

   ```bash
   npm start        # Expo dev server + QR for Expo Go
   npm run ios      # iOS simulator (needs a Mac)
   npm run android  # Android emulator/device
   npm run web      # browser
   ```

App source lives in `src/app/` (screens & routes) and `src/components/`
(shared UI). Adding a file under `src/app/` adds a route.
